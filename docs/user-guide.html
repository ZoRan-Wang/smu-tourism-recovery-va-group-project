User Guide

Purpose

This guide explains how to use the final tourism analytics app and, in particular, how to work through the Forecasting module as an interactive analytical workflow instead of a static chart page.

App Structure

The live Shiny app is organised into four top-level tabs:

  1. Time Series Visual Analysis
  2. Time Series Clustering
  3. Forecasting
  4. About

The intended reading order is:

  1. use Time Series Visual Analysis to inspect the raw series and metadata,
  2. use Time Series Clustering to compare country recovery trajectories,
  3. use Forecasting to build and interpret a forward projection for one country-level arrivals series.

Shared Data Scope

The app uses two coordinated data layers:

  • Shared arrivals backbone: data/raw/visitor_arrivals_full_dataset.xlsx
  • Shared processed series: data/processed/arrivals_country_long.csv, data/processed/arrivals_country_wide.csv
  • Shared clustering artifacts: data/processed/clustering_country_wide.csv, data/processed/clustering_country_long.csv, data/processed/clustering_series_metadata.csv
  • Optional supporting tourism context: data/raw/tourism_update.xlsx

Country-level visitor arrivals remain the common analytical target across explorer, clustering, and forecasting. Hotel occupancy, average length of stay, number of hotels, and total room revenue are used only as optional supporting context for interpretation.

Step 1: Explore the Series

Open the Time Series Visual Analysis tab to:

  1. choose a monthly tourism series such as Visitor Arrivals: China,
  2. inspect the recent trend and volatility,
  3. confirm the source and unit metadata before moving to deeper analysis.

This tab is intentionally lightweight. It is designed to help users confirm what the target series looks like before they move into clustering or forecasting.

Step 2: Compare Recovery Trajectories

Open the Time Series Clustering tab to:

  1. select a focused set of country-level visitor-arrivals series,
  2. set the year window and choose the normalization mode,
  3. pick a working k and run the clustering,
  4. review the module in order: Dashboard, Pattern Explorer, Focus Market in Context, Assignments.

The clustering unit is a country trajectory, not an individual month. This module answers which markets recovered in similar ways over time.

Step 3: Use the Forecasting Module

Open the Forecasting tab when you want to project one country-level arrivals series forward and compare forecasting models on a time-aware holdout window.

3.1 Configure the controls

Use the left control panel to choose:

  1. Country arrival series
    Select the market to forecast, for example Visitor Arrivals: China or Visitor Arrivals: Philippines.

  2. Forecast engine
    Three engine modes are available:

    • Auto (best available): uses the full modeltime workflow when the package stack is ready; otherwise falls back to the lighter forecast implementation.
    • Require modeltime: runs only the modeltime workflow. If the required packages are missing, the module reports the failure in the status panel instead of silently falling back.
    • Use lightweight fallback: always uses the lighter forecast implementation.

  3. Test / forecast horizon
    This controls how many trailing months are held out as the test window and how many months are projected forward.

  4. Models to compare
    You can compare:

    • Seasonal Naive
    • ETS
    • ARIMA

  5. Context indicators
    These optional indicators are used in the context view:

    • Hotel occupancy
    • Average stay length
    • Room revenue

  6. Model ranking metric
    Choose which metric decides the winning model:

    • RMSE
    • MAE
    • MAPE

After setting the controls, click Run Forecasting. The module does not auto-run on page load.

3.2 Forecast tab

The Forecast tab is the main results page. It contains four panels:

Holdout comparison

  • compares the selected models on the unseen test window,
  • helps you judge whether the model beats the seasonal baseline on recent months,
  • should be read before trusting the forward forecast.

Forward forecast

  • projects the selected country series into future months,
  • always uses the current best holdout model under the chosen ranking metric,
  • updates when you change the ranking metric or the set of compared models.

Accuracy summary

  • provides a text summary of each model’s RMSE, MAE, and MAPE,
  • ranks the models by the metric chosen in the control panel,
  • makes it easier to compare models without reading a dense data table.

Series summary

  • reports the observation count,
  • reports the date coverage,
  • confirms that the scope is country-level arrivals,
  • lists the models currently included in the run.

3.3 Model Studio tab

The Model Studio tab is designed to explain why one model wins.

It contains:

Model leaderboard

  • ranks candidate models by the selected metric,
  • lets you switch between RMSE, MAE, and MAPE,
  • shows how the ranking changes under different evaluation rules.

Holdout residual comparison

  • plots actual - predicted residuals on the holdout window,
  • helps identify which model stays closer to zero,
  • makes month-to-month miss patterns easier to see.

Engine status

  • shows the requested engine,
  • shows the workflow that actually executed,
  • reports whether fallback and modeltime are available,
  • reports missing packages when modeltime is unavailable,
  • explains strict failure when Require modeltime is selected.

Model interpretation

  • identifies the current winning model,
  • explains the gap to the next candidate under the selected metric,
  • reminds users how to interpret Seasonal Naive, ETS, and ARIMA.

3.4 Diagnostics tab

The Diagnostics tab focuses on the structure of the selected time series.

It includes:

Raw time series

  • shows the full historical arrivals path,
  • helps identify structural breaks such as the COVID shock and recovery period.

Seasonal pattern

  • compares the month-to-month pattern across years,
  • helps determine whether the series has stable seasonal behaviour.

Decomposition

  • breaks the series into trend, seasonal, and remainder components,
  • supports interpretation of why the forecast behaves the way it does.

Split summary

  • reports how the training and testing windows were divided,
  • confirms the date span and observation counts for each segment.

3.5 Context tab

The Context tab connects the arrivals forecast to broader tourism performance.

It includes:

Tourism performance context

  • overlays the selected arrivals series with hotel occupancy, stay length, and room revenue,
  • normalizes the series so turning points and relative movement can be compared,
  • supports business interpretation rather than direct value comparison.

Interpretation guide

This guide reminds users to read the module in the following order:

  1. start from Holdout comparison to judge model credibility,
  2. move to Model Studio to understand ranking and residual behaviour,
  3. use Diagnostics to interpret the series structure,
  4. return to Context before making business conclusions.

Interpretation Tips

  1. Do not start from the forward forecast alone. Read the holdout comparison first.
  2. Use the selected ranking metric consistently when discussing the best model.
  3. Use diagnostics to explain whether the winning model is driven by trend, seasonality, or irregular variation.
  4. Use the context chart to connect forecast movement with tourism business indicators, rather than treating arrivals in isolation.

Recommended Presentation Flow

When demonstrating the app:

  1. choose one country series in the control panel,
  2. set the horizon and model list,
  3. click Run Forecasting,
  4. explain the Forecast tab first,
  5. use Model Studio to justify the chosen model,
  6. use Diagnostics to explain the time-series structure,
  7. finish with Context to link the forecast to tourism performance.

Summary

The forecasting module is intended to do more than display a future trend line. It combines:

  • benchmark and model-based forecasting,
  • model comparison,
  • residual diagnostics,
  • structural decomposition,
  • and business-context interpretation

within one Shiny workflow built on the shared country-arrivals backbone.
